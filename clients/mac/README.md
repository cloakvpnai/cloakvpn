# Lattice VPN — macOS client

Native macOS menu-bar VPN app, sharing the post-quantum tunnel core with the iOS app.

## What's in this directory

```

├── project.yml                         xcodegen spec (the source of truth)
├── LatticeMac.xcodeproj/               generated Xcode project (committed)
├── LatticeMac/                         host app (menu bar utility)
│   ├── LatticeMacApp.swift             SwiftUI @main + Settings scene
│   ├── AppDelegate.swift               NSStatusItem + popover lifecycle
│   ├── MenuBar/
│   │   ├── MenuBarController.swift     status-item / popover / context-menu
│   │   ├── ConnectionPopoverView.swift the dropdown UI
│   │   └── StatusItemIcon.swift        SF Symbol per connection state
│   ├── Settings/

│   ├── ViewModels/
│   │   └── ConnectionViewModel.swift   bridges UI <-> TunnelManager
│   └── Resources/
│       ├── Info.plist                  GENERATED by xcodegen from project.yml
│       └── LatticeMac.entitlements     GENERATED by xcodegen from project.yml
├── LatticeMacTunnel/                   NEPacketTunnelProvider system extension
```

The shared core (TunnelManager, RosenpassBridge, rosenpassffi, ConfigParser,
Region, AppGroupKeyStore, CloakAuthClient, SubscriptionInfo, RosenpassDriver,
PacketTunnelProvider) lives in `../ios/` and is referenced via project.yml's
`sources:` lists — no file duplication.

## Building

```bash
cd clients/mac
xcodegen generate                # regenerate Xcode project from project.yml
xcodebuild -scheme LatticeMac \
           -configuration Debug \
           -destination 'platform=macOS' \
           build
```

Or open `LatticeMac.xcodeproj` in Xcode and hit Cmd-R.

## Quirks worth knowing

### 1. Why is `rosenpassffiFFI.framework` separate from the iOS xcframework?

The iOS app links a static `RosenpassFFI.xcframework` at
`../ios/RosenpassFFI.xcframework`. The Mac app instead links a *dynamic*
framework at `Frameworks/rosenpassffiFFI.framework`. Two reasons:

1. **Duplicate-symbol conflict at link time.** Mullvad's `libmaybenot`
   (a WireGuardKit dep used for traffic obfuscation) is also Rust-based
   and ships its own copy of Rust `std`, including a definition of
   `_rust_eh_personality`. A static-lib Mac slice of rosenpassffi would
   duplicate that definition; a dynamic framework hides it.

2. **An xcframework can't mix library types.** Apple's `-create-xcframework`
   rejects bundling iOS-static slices and a Mac-dynamic slice together.

So we ship the iOS xcframework (static, contains `ios-arm64` +
`ios-arm64-simulator`) AND the Mac framework (dynamic, `macos-arm64` only)
as two separate artifacts.

### 2. How is the Rust shared between iOS and Mac?

Both come from the same `clients/ios/RosenpassFFI/` crate. Build:

- **iOS**: `clients/ios/RosenpassFFI/build-xcframework.sh` →
  `clients/ios/RosenpassFFI.xcframework` (static, iOS-only)
- **Mac**: `clients/mac/Scripts/build-mac-framework.sh` →
  `clients/mac/Frameworks/rosenpassffiFFI.framework` (dynamic)

Both consume `target/aarch64-apple-{ios,darwin}/release/librosenpassffi.{a,dylib}`,
so a single `cargo build` per target produces both.

### 3. Why is `Cargo.toml`'s release profile set to `panic = "abort"`?

Eliminates panic-unwinding code paths from the library, which:
- Prevents undefined behavior if a Rust panic propagated through C FFI
  back into Swift (Swift can't unwind Rust frames).
- Reduces binary size ~10-15%.
- Is the iOS default anyway (iOS targets force panic=abort).

The `_rust_eh_personality` symbol still gets emitted by the precompiled
Mac std (we can't easily rebuild std without nightly Rust), but it's
contained inside the dynamic framework — not exposed at link time.

### 4. Why does `PacketTunnelProvider` no longer have `final`?

So the Mac target's `LatticeMacTunnelProvider` can subclass it to add
Direct-Download-build-specific overrides (e.g. diagnostic log flush on
extension stop). Runtime cost is negligible (one v-table dispatch on
methods that aren't hot).

### 5. Why are Info.plist and entitlements files committed if they're generated?

Two reasons: (a) so a fresh checkout opens cleanly in Xcode without
needing xcodegen installed, and (b) so Xcode's "Signing & Capabilities"
tab can read them. They're regenerated from `project.yml` each time
`xcodegen generate` runs.

**Don't hand-edit them** — edit `project.yml` under `info.properties`
or `entitlements.properties`, then regenerate.

## Distribution (after LLC migration)

### Mac App Store

1. Product → Archive (with scheme "LatticeMac (App Store)")
2. Window → Organizer → archive → Distribute App → App Store Connect

### Direct Download (Developer ID + notarized DMG)

1. Product → Archive (with scheme "LatticeMac (Direct Download)")
2. Export with Developer ID signing
3. Run `Scripts/notarize.sh path/to/LatticeMac.app`
4. The script staples + produces a `.dmg` ready to host at
   `https://latticevpn.ai/download/latticemac.dmg`

Notarization requires app-specific password stored in the keychain:
```bash
xcrun notarytool store-credentials latticevpn-notary \
    --apple-id demetris@neuroaistudios.com \
    --team-id <TEAM_ID> \
    --password <APP_SPECIFIC_PASSWORD>
```
