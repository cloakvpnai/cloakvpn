## The big win this session

**The 19-hour debugging marathon's root cause is solved and committed.**

The previous session's "iPhone NE doesn't carry traffic" bug was caused by our SwiftPM dependency on **mullvad/wireguard-apple** (mullvad-master branch). Mullvad's `WireGuardAdapter.start()` was rewritten to support multi-hop / DAITA and in the process **removed the `setTunnelNetworkSettings` call** that upstream wireguard-apple makes. Without that call, iOS never installs IPs/DNS/routes on the utun, so wireguard-go runs blind: WG handshake completes (UDP, doesn't traverse utun), keepalives flow (internal heartbeat), but no app traffic ever enters utun.

We **can't** switch SwiftPM dep to upstream WireGuard/wireguard-apple because upstream's `Package.swift` has been broken since 2023 (declares `swift-tools-version:5.3` but uses `.macOS(.v12)`/`.iOS(.v15)` which require 5.5+). SwiftPM correctly rejects every tagged release.

**Fix shipped:** patch our PacketTunnelProvider to call `setTunnelNetworkSettings` ourselves before `adapter.start`. Plus a `makeNetworkSettings(from:)` helper that builds `NEPacketTunnelNetworkSettings` from a `TunnelConfiguration` (port of upstream's `PacketTunnelSettingsGenerator.generateNetworkSettings` logic, kept identical so routing/DNS matches the official WG iOS app).

**Validated end-to-end on iPhone:** test-ipv6.com showed `5.78.203.171`. The custom NE works.

**Commit:** `514d4ab fix(ios): call setTunnelNetworkSettings ourselves to compensate for Mullvad fork` on branch `wg-upstream-fork`, pushed to origin.
**Triage report:** `docs/TRIAGE_2026-04-27.md` (~5 pages, full root-cause analysis).

This makes the previous handoff's "Tomorrow's plan: Option 2 ŌĆö fork upstream WG iOS" **obsolete**. We don't need to fork upstream. The SwiftPM dep stays on Mullvad's fork, and our PacketTunnelProvider compensates for the missing call.

The Rust FFI fix was applied to `clients/ios/RosenpassFFI/src/lib.rs` `handle_message` to send InitConf bytes BEFORE returning DerivedPsk.

**All this PQC source work is preserved in `stash@{0}` ("PQC work in progress before revert"). Recoverable via `git stash apply stash@{0}`.**

We then rebuilt the RosenpassFFI xcframework via `./build-xcframework.sh` (rust toolchain 1.88.0). The build succeeded. Binary copied into `RosenpassFFI.xcframework/`. Then in Xcode: Clean Build Folder + Run.

**Result: app installs but crashes immediately on launch.** Icon disappears from home screen (iOS auto-hides after repeated crash-on-launch), only appears in App Switcher recents.

Diagnosis attempts that came up empty:
- Crash logs not in `~/Library/Logs/DiagnosticReports/` or any standard location
- Xcode's debug area showed nothing during the brief launch window
- xcresult files contain only build logs (no runtime data ŌĆö runtime crashes don't get captured here)
- `xcrun devicectl list apps` failed (`No provider was found`, syntax also drifted)
- `idevicesyslog` ran but produced no captured output (likely needs `idevicepair pair` first; not yet attempted)
- Console.app streaming was set up but the user didn't get a useful capture before fatigue hit

**Critical test we ran:** hard-reset source to `514d4ab` (the known-working WG commit), then rebuilt FFI from that exact unchanged source. The rebuilt binary STILL crashes. **This proves the crash isn't from any source change in this session ŌĆö it's environmental.**

Likely culprits for the environmental crash:
1. **rust toolchain version drift.** The previous session built with whatever rust was current then. We just installed 1.88.0 fresh via `rustup-init`. The build script pins to 1.88.0 but the previous binary may have been built against a different version ŌĆö there could be a uniffi runtime/binding ABI drift across rust versions.
2. **uniffi version pinning.** `Cargo.toml` says `uniffi = { version = "0.27" }` (any 0.27.x). If the bindings were generated by 0.27.4 and the binary built with 0.27.5 (or vice versa), the contract version checks (`bindings_contract_version = 26` in rosenpassffi.swift line 905) might fire `fatalError("UniFFI contract version mismatch")` immediately on first FFI call.
3. **Debug vs release build.** Our build was `--debug` (149 MB static archive). The previous binary may have been `--release` (smaller, more optimized, different code paths).
4. **Cargo.lock drift on transitive deps** ŌĆö rosenpass and its deps are pinned to git rev `b096cb1` but transitive crates (cargo registry deps) get whatever Cargo.lock allows.
