## Root cause

The `CloakTunnel` NetworkExtension target was depending on **Mullvad's fork** of `wireguard-apple`, not the official upstream:

```
repositoryURL = "https://github.com/mullvad/wireguard-apple.git"
requirement = { branch = "mullvad-master"; kind = branch; }
```

Mullvad's fork has rewritten `WireGuardAdapter.start()` to support multi-hop and DAITA (their traffic-pattern obfuscation). In the rewrite they **removed the call to `setTunnelNetworkSettings`** that upstream performs in `start()`. The diff:

```diff
 public func start(tunnelConfiguration: TunnelConfiguration, completionHandler: ...) {
     workQueue.async {
         ŌĆ”
         do {
             let settingsGenerator = try self.makeSettingsGenerator(with: tunnelConfiguration)
-            try self.setNetworkSettings(settingsGenerator.generateNetworkSettings())   // ŌåÉ upstream
+            // (call removed in mullvad-master)

             let (wgConfig, resolutionResults) = settingsGenerator.uapiConfiguration()
             ŌĆ”
             self.state = .started(
-                try self.startWireGuardBackend(wgConfig: wgConfig),                     // ŌåÉ upstream
+                try self.startWireGuardBackend(exitWgConfig:..., privateAddress:...,    // ŌåÉ mullvad
+                                               entryWgConfig:..., mtu: 1280, daita:...),
                 settingsGenerator
             )
```

In Mullvad's real product they call `setTunnelNetworkSettings` from a higher-level multi-hop coordinator that wasn't shipped in their public OSS repo's reference `PacketTunnelProvider`. They aren't trying to ship a drop-in upstream-compatible package; they ship the underlying machinery their own app builds on top of.

Cloak VPN's `PacketTunnelProvider` was written to match upstream's contract ŌĆö it does `adapter.start(tunnelConfiguration:)` and trusts the adapter to wire up the OS network settings. Against Mullvad's fork that contract is violated, so:

- `wgTurnOn` runs and hands the wireguard-go data plane a `tunnelFileDescriptor`.
- The tunnel file descriptor exists *only because* iOS pre-allocates a utun for an NE process. But without `setTunnelNetworkSettings`, **iOS never installs any IPs, DNS, or routes on that utun**, so application traffic is never routed into it.
- WG handshake completes (UDP, doesn't traverse utun).
- Keepalives fire every 25s (internal to wireguard-go's heartbeat loop).
- All Safari/app traffic continues over the cellular/Wi-Fi default route ŌĆö exactly the symptom: `test-ipv6.com` shows the NAT IP, not the server IP. With `includeAllNetworks = true`, traffic has nowhere to go and Safari reports "iPhone is not connected to the internet."

This explains every symptom catalogued in the 2026-04-27 03:00 HST handoff:

| Symptom from handoff | Why it happens with Mullvad fork bug |
|---|---|
| `nm CloakTunnel.debug.dylib \| grep wgTurnOn` shows symbols | wireguard-go *is* linked and runs; that's not where the failure is. |
| Handshake completes, keepalives flow | UDP at the wire layer; doesn't depend on utun routing. |
| Safari: "iPhone is not connected to the internet" with `includeAllNetworks=true` | iOS has no routes installed on utun, full-tunnel mode has nowhere to send packets. |
| Without `includeAllNetworks`, traffic leaks via ISP | Same reason: iOS doesn't know utun is the route to 0.0.0.0/0, so it falls back to default. |
| Removing the previous `setTunnelNetworkSettings` *override* didn't help | Removing an override doesn't add a missing call. The previous Claude correctly noticed that "upstream doesn't override," but missed that "upstream's *adapter* calls it for you, and Mullvad's doesn't." |
| Replacing PacketTunnelProvider with code structurally identical to upstream's didn't help | Source-code structure was never the bug ŌĆö the Swift package backing the import was. |

## Why the previous session missed it

The previous Claude correctly enumerated and ruled out everything *they could see in the source*. They never compared the **package-resolution layer** against upstream because the Swift import (`import WireGuardKit`) is identical in both. Mullvad rebrands their fork's package as `wireguard-apple` and exports `WireGuardKit` with the same module name ŌĆö the import line gives no hint that you're not on upstream.

The clue was buried in `project.pbxproj` (one line) and in the secondary build artifact `libmaybenot.a` (which only Mullvad's WireGuardKitGo Makefile produces).

---

## Fix applied (this session)

Two surgical edits to `clients/ios/CloakVPN.xcodeproj/project.pbxproj`:

1. **Switch SwiftPM dependency to upstream, pinned to a tagged release:**

   ```diff
   -repositoryURL = "https://github.com/mullvad/wireguard-apple.git";
   -requirement = { branch = "mullvad-master"; kind = branch; };
   +repositoryURL = "https://github.com/WireGuard/wireguard-apple.git";
   +requirement = { kind = exactVersion; version = "1.0.16-27"; };
   ```

   `1.0.16-27` corresponds to upstream commit `2fec12a6e1f6e3460b6ee483aa00ad29cddadab1`, the latest tagged release at the time of this triage. Pinning to an exact version (rather than a moving branch) gives us reproducibility ŌĆö no future change in upstream can silently break our build.

2. **Remove `libmaybenot.a` from the `Build wg-go bridge` phase outputPaths.** Upstream's WireGuardKitGo Makefile only produces `libwg-go.a`; `libmaybenot.a` was a Mullvad-only artifact (their DAITA traffic obfuscation library) and nothing in our project consumed it.

No Swift source code changes were required. Our `PacketTunnelProvider`, `RosenpassBridge`, `TunnelManager`, and `ConfigParser` already use the public `WireGuardKit` API surface that upstream provides (`WireGuardAdapter`, `TunnelConfiguration`, `InterfaceConfiguration`, `PeerConfiguration`, `PrivateKey`, `PublicKey`, `PreSharedKey`, `IPAddressRange`, `Endpoint`, `DNSServer`, `WireGuardLogLevel`). All confirmed present in upstream `1.0.16-27`.

## Things that were red herrings (do not retry)

Every item the previous Claude listed as a dead-end is *still* a dead-end. The fix had nothing to do with:

- `setTunnelNetworkSettings` overrides on PacketTunnelProvider (the previous removal was harmless once we're on upstream)
- `enforceRoutes`, `excludeXxx` flags
- `WireGuardKitGo` linkage (it was always linked correctly)
- `makeTunnelConfiguration` byte-equivalence (it was always equivalent)
- DerivedData cleans, app reinstalls, iPhone reboots
- Apple Developer Profile or signing identity

The only thing that wasn't checked was the SwiftPM dependency. That was the bug.

## Risks of this fix

- **`libwg-go.a` Go-runtime divergence.** Upstream and Mullvad both vendor wireguard-go inside WireGuardKitGo, but at potentially different commits. If your test on Step 2 fails with a `wgTurnOn` error or stalls during handshake, that would point at a Go-side regression ŌĆö but the most likely failure mode here is "build succeeds, traffic flows." Mullvad's main divergence in WireGuardKitGo is additive (in-tunnel TCP, ICMP, multi-hop, DAITA) ŌĆö none of which we use.
- **Lost feature regression.** We were never using maybenot, multi-hop, in-tunnel TCP/ICMP, or any other Mullvad-specific feature, so dropping them costs nothing.
- **Tag pinning means manual updates later.** If a CVE drops in upstream wireguard-apple, we'll have to bump `version = "1.0.16-27"` manually. That's the right trade-off for reproducibility.
